# 1K Builder — SPEC (Source of Truth)

> **문서 성격**: OpenSpec 방식의 단일 living spec. 구현 들어가기 전·도중에 팀(3인)과 AI 코딩 어시스턴트(Claude Code 등)가 **이 문서 하나만 읽고도 기획 방향을 잃지 않도록** 하는 권위 있는 참조 문서.
> **읽는 순서**: ① Proposal(왜) → ② Design 결정(어떻게) → ③ Requirements(무엇을) → ④ KPI → ⑤ Out of Scope → ⑥ 결정 대기 → ⑦ Decision Log.
> **버전**: v1.0 · 한국 우선 MVP

---

## 0. 용어

- **채널**: 서비스가 운영하는 사용자의 인스타 비즈니스 계정 1개.
- **컨셉 잠금**: 페르소나·톤·비주얼·콘텐츠 기둥·게시 주기를 1회 확정하는 온보딩 단계.
- **시안**: AI가 생성한 게시물/릴스 초안(검수 전).
- **오토파일럿**: 신뢰 누적 후 저위험 콘텐츠를 사람 검수 없이 흐르게 하는 모드.
- **3-레이어**: 생성(Claude) / 발행·데이터(Meta Graph API) / 오케스트레이션(n8n).

---

## 1. Proposal — 왜 만드는가

### 1.1 문제
예비 창작자는 "내 주제로 수익화할 인스타"를 원하지만, 매일 기획·제작·발행할 시간·역량이 없어 콜드스타트(0→1K) 구간에서 한 달 내 포기한다. 1,000 팔로워는 협찬·수익화가 시작되는 가장 어려운 진입선이다.

### 1.2 핵심 인사이트 (타당성 검증 결과)
- **자동 게시 자체는 차별점이 아니다.** n8n + Graph API 자동 발행은 이미 흔하다(공개 템플릿 다수). 병목은 "어떻게 올리지"가 아니라 "어떻게 *참여도 높은* 팔로워를 키우지"다.
- **플랫폼이 역풍이다.** 인스타는 2026년 자동화·템플릿 콘텐츠 도달을 낮추고, AI 라벨을 강제하며, 인간 크리에이터·진정성에 가중치를 둔다. → 완전 자동 양산형은 알고리즘이 직접 누른다.
- **따라서 설계 방향**: "완전 자동 공장"이 아니라 **사람 손을 거치는 코파일럿**으로 포지셔닝하고, 자동화가 아니라 *성장 결과(팔로워 순증)*를 가치로 판다. 검수·AI라벨을 약점이 아니라 신뢰·채택 장치로 쓴다.

### 1.3 목표
콜드스타트를 자동 운영으로 통과시켜 채널을 1K까지 도달시키고, 1K 이후 지속 가능한 수익화로 잇는다.

## 2. Design 결정 (Architecture & Key Decisions)

### 2.1 3-레이어 분리 (핵심 결정)
| 레이어 | 책임 | 도구 |
|---|---|---|
| 생성 | 기획·카피·스크립트 등 **언어 가공만** | Claude API |
| 발행·데이터 | 인스타 게시·성과 수집 | Meta Graph API (앱 검수 필수) |
| 오케스트레이션 | 예약·발행·웹훅·재시도 | n8n + 자체 스케줄러 |
| 영상 | 릴스·숏츠 렌더 | ZapCap·Creatomate 등 |

**결정 근거**: Claude API로는 인스타 직접 접근 불가. 세 레이어를 분리해야 각 레이어를 독립적으로 교체·확장할 수 있다.

### 2.2 인증/데이터
- IG 비즈니스 계정 **OAuth 연결**. 비밀번호 미보관, 토큰 암호화 저장.
- 콘텐츠·성과는 Graph API로만 수집(follower_count, follows_and_unfollows, reach, comments 등).

### 2.3 발행 메커니즘
- Graph API 2단계 발행(컨테이너 생성 → FINISHED 확인 → 게시) 준수.
- 네이티브 예약이 없으므로 자체 스케줄러로 예약 시각에 n8n 워크플로 트리거.

### 2.4 콘텐츠 엔진 = 3모드 (한 엔진, 입력만 다름)
- **생성형(A)**: 지식만 있고 소스 없음 → 0에서 콘텐츠 생성.
- **자산 기반(B)**: 제품·매장 자산 → 콘텐츠로 가공.
- **증강(C)**: 보유 영상 → 숏츠로 편집·증강. *(완전 생성은 M2)*

## 3. Requirements (Capabilities)

> 형식: `Requirement`(시스템이 반드시 보장해야 하는 것) + `Scenario`(WHEN/THEN). 구현·테스트의 기준.

### 3.1 Capability: 온보딩 & 컨셉 잠금

#### Requirement: 컨셉 잠금
시스템은 사용자가 주제·타깃을 입력하면 채널 페르소나·톤·비주얼·콘텐츠 기둥·게시 주기를 제안하고, 사용자가 1회 확정(잠금)하도록 해야 한다(SHALL). 잠긴 가드레일은 이후 모든 시안 생성에 자동 상속된다.

##### Scenario: 사용자가 컨셉을 확정
- WHEN 사용자가 주제·타깃을 입력하고 제안된 컨셉을 승인
- THEN 시스템은 페르소나·톤·비주얼·콘텐츠 기둥·주기를 채널 설정으로 저장
- AND 이후 생성되는 시안은 이 설정을 제약으로 사용

#### Requirement: IG 계정 연동
시스템은 사용자의 IG 비즈니스 계정을 OAuth로 연동해야 하며(SHALL), 비밀번호를 저장하지 않고 토큰을 암호화 보관해야 한다.

##### Scenario: 계정 연동 실패
- WHEN 비즈니스 계정 전환·페이지 연결 등 전제 조건이 미충족
- THEN 시스템은 무엇이 필요한지 안내하고 발행 기능을 잠금 상태로 유지

### 3.2 Capability: 콘텐츠 생성 엔진 (3모드)

#### Requirement: 시안 생성
시스템은 잠긴 컨셉과 선택된 모드(생성형/자산기반/증강)에 따라 전략·카드뉴스(템플릿+브랜드 컬러)·릴스 스크립트를 자동 생성해야 한다(SHALL). 카드뉴스가 메인, 릴스가 보조다.

##### Scenario: 주제 추천·탐색
- WHEN 사용자가 무엇을 올릴지 모름
- THEN 시스템은 핫/팔로워 잘 모이는/조회수 높은 주제를 추천하고 전략을 자동 생성

### 3.3 Capability: 휴먼 인 더 루프 검수 (차별화 핵심)

> 대원칙: **검수 = 결정이지 작업이 아니다.** 사용자는 만들지 않는다 — 고르거나, 거부하거나, 말로 지시할 뿐이다.

#### Requirement: 3단계 검수
시스템은 ① 컨셉 잠금(1회·무겁게) ② 계획 승인(주기적·중간) ③ 게시물 검수(매번·가볍게)의 3단계 검수를 제공해야 한다(SHALL). 위층일수록 무겁고 드물고, 아래층일수록 가볍고 잦다.

#### Requirement: 게시물 검수 효율
시스템은 게시물 검수를 5초 결정으로 만들기 위해 다음을 제공해야 한다(SHALL):
1. AI 자가 점검 — 의심 지점(미검증 주장·이미지 아티팩트 등)을 먼저 플래그하여, 빈 검수가 아니라 플래그된 항목만 판단하게 한다.
2. 자연어 수정 — 직접 편집이 아니라 "더 캐주얼하게" 등 지시로 재생성.
3. 변형안 선택 — 2~3개 중 선택.

##### Scenario: 사용자가 시안을 수정 요청
- WHEN 사용자가 "더 캐주얼하게"라고 입력
- THEN 시스템은 캡션 에디터를 열지 않고 톤을 바꿔 재생성

#### Requirement: 리스크 등급별 라우팅
시스템은 시안을 저위험/고위험으로 분류해야 하며(SHALL), 고위험(미검증 사실·실존 인물·브랜드/효능·규제 업종 등)은 사람 검수를 강제하고 체크박스 확인을 받아야 한다. 저위험은 원탭 승인 또는 조건부 자동 흐름을 허용한다.

##### Scenario: 고위험 게시물
- WHEN 시안이 고위험으로 분류
- THEN 시스템은 사람 검수 없이는 발행하지 않으며(MUST NOT), 명시 체크박스 확인을 요구

#### Requirement: 오토파일럿 (신뢰 단계 상승)
시스템은 신뢰 누적 후(예: 동일 저위험 유형 10회 연속 승인) 저위험 콘텐츠의 자동 흐름을 제안할 수 있다(MAY). 단 킬 스위치와 사후 검수 피드(게시 후 소급 삭제)를 상시 유지해야 한다(SHALL).

### 3.4 Capability: 발행 (n8n + Graph API)

#### Requirement: 승인 후 발행
시스템은 검수 기본 ON 상태에서, 승인된 시안만 발행해야 한다(SHALL). 발행은 Graph API 2단계(컨테이너 → FINISHED → 게시)를 거친다.

##### Scenario: 예약 발행
- WHEN 예약 시각 도달
- THEN 자체 스케줄러가 n8n 워크플로를 트리거하고, 컨테이너가 FINISHED가 된 뒤 게시

### 3.5 Capability: 팔로워 수집 (코멘트→DM 리드마그넷)

#### Requirement: 코멘트→DM 리드마그넷
시스템은 공식 Messaging/Comments API로 리드마그넷을 운영해야 한다(SHALL): 특정 댓글 트리거 → 정보 DM 발송. 자동팔로우·콜드 DM은 사용하지 않는다(MUST NOT).

##### Scenario: 댓글 트리거
- WHEN 사용자가 지정 키워드를 댓글로 남김
- THEN 시스템은 공식 API로 해당 사용자에게 정보 DM을 발송하고 자체 DM 로그에 기록

### 3.6 Capability: 분석 & KPI 계측

#### Requirement: 성과 대시보드 & 이벤트 로깅
시스템은 발행·승인·연동·도달·팔로우 순증·리드 이벤트를 로깅하고(SHALL), KPI(4장) 산출에 필요한 데이터를 제공해야 한다. Aha moment("내가 안 했는데 올라갔다")는 첫 자동 발행이며, 활성화 이벤트로 기록한다.

### 3.7 Capability: 컴플라이언스 & 안전

#### Requirement: AI 라벨 & 감사 추적
시스템은 AI 생성 콘텐츠에 AI 라벨을 기본 적용해야 하며(SHALL), 누가·언제·무엇을 승인했는지 승인 로그를 남겨야 한다(감사 추적).

#### Requirement: 도메인 컴플라이언스
시스템은 도메인 규칙을 검수 플래그/고위험 등급에 주입해야 한다(SHALL): 금융=권유 표현 차단, 패션·소상공인=협찬·뒷광고 자동 표기.

> ⚠️ 출시 전 법 해석 확인: 오토파일럿(콘텐츠 유형 사전 승인)이 EU AI Act §50의 '사람 검토'로 인정되는지는 회색지대. 자동 흐름은 '명백한 저위험 + 라벨 부착'으로 좁게 두고 자문 확정.

## 4. KPI (전체)

**미션 지표(장기·가치 실현):** 1,000 팔로워 도달 계정 수. *주간 운영 관리엔 미사용(후행).*

**운영 North Star(주간):** 주간 활성 채널당 팔로워 순증.
- 분해: `팔로워 순증 ≈ 발행 수 × 게시물당 도달 × (도달→팔로우 전환율)`
- 마일스톤(가설): 1개월 +100 / 3개월 누적 500 / 6개월 1,000

**퍼널(서비스 데이터)**
| 단계 | 지표 | 정의 | 목표(가설) |
|---|---|---|---|
| Acquisition | 가입자 수 / IG 연동 완료율 | 신규 회원 / 연동 완료 비율 | — / 70%+ |
| Activation | 활성화율 | 48h 내 첫 승인·발행 도달률 | 40%+ |
| 습관 형성 | 2회차 발행률(7일 내) | 첫 발행 후 7일 내 2번째 승인·발행 도달 비율 | 60% |
| Retention | 5건 코호트 잔존율 | 자동발행 5건 도달 코호트 이후 잔존율 | 측정 후 |
| Retention | W4 발행 유지율(headline) | 4주차에도 발행 중인 비율 | 25%+ |

**성장 입력(Graph API 데이터):** 게시물당 유입 대비 이탈(순증 효율) · 주간 리드마그넷 유입.

**가드레일(동시 감시)**
| 지표 | 목표 |
|---|---|
| 게시물당 참여율 (NSM 필수 짝) | 벤치마크 대비 하락 없음 |
| 무수정 승인율 | 건강구간 60~85% |
| 평균 검수 소요 시간 | < 1~2분/건 |
| 계정 제재·셰도우밴율 | < 1% (목표 0) |
| 고위험 검수 누락률 | 0% (하드 게이트) |

**가치 증명:** 연속 발행 회원 vs 중단 회원의 1K 도달 속도 비교 = 성장 리프트.

## 7. Decision Log (왜 이렇게 정했나 — 재질문 방지용)

- **North Star를 '발행 수'나 '1K'가 아니라 '주간 팔로워 순증'으로**: 1K는 후행·vague해 운영 불가(미션으로만). 발행 수는 도달이 죽어도 오르는 헛 지표 위험. 팔로워 순증이 통제 가능하면서 1K를 직접 예측.
- **습관 지표를 '연속 발행 회원 수' → '2회차 발행률(7일)'로 교체**: 스트릭 정의 모호 + 5건 코호트와 중복. 2회차 비율이 목표 설정·해석이 쉽고 이탈자를 분모로 자연 처리. (median TTSP는 데이터 쌓인 뒤 진단 보조.)
- **완전 자동이 아닌 휴먼 인 더 루프**: 자동 양산형은 2026 인스타 알고리즘이 누름 + 규제·신뢰 문제. 검수를 채택·신뢰 장치로 전환.
- **3-레이어 분리**: Claude API로 인스타 직접 접근 불가. 생성/발행/오케스트레이션 독립 교체 위해.
- **공식 API만**: 자동팔로우·콜드DM은 계정 정지 직결. 성장은 코멘트→DM 등 합법 레버로만.
- **카드뉴스 메인 + 릴스 보조**: MVP 제작 난이도·속도 고려. 숏츠 완전 생성은 M2로 미룸.
